🔍 SearchAgent — 完整搜索管线（735行）

S
SearchAgent 活跃 · 8 tools

独立于 AgentService 的完整搜索管线。双模式：快速管线（search）+ ReAct 复杂查询（search_react）。

快速管线：search() 8 步

  1. 1
    LLM 提取筛选条件
    EXTRACT_FILTERS_PROMPT + complete_json(t=0.0)。设施映射 17 条规则。失败时返回 {}。
  2. 2
    学校/机构查找 _lookup_institution()
    4 级匹配：exact abbreviation → ILIKE name/cn → aliases[] 包含 → ILIKE abbreviation。查 universities 表。
  3. 3
    搜索（双分支）
    有 institute_id → _geo_search()：Haversine 精筛，半径 3→5→10km 渐进放宽
    无 institute_id → _search_with_relaxation()：district→type→bedrooms→price×1.2 逐级放宽
  4. 4
    (可选) 通勤过滤 _filter_by_commute()
    多引擎降级链：主API→备选→ORS→Haversine。4 级时间放宽（max→2×→3×→4×）
  5. 5
    得分间隙检测 detect_score_gap()
    检测 Top 1 与后续得分的断层，用于安全兜底判断
  6. 6
    安全兜底 SafeFallback
    候选太少/得分断层/放宽过多 → 生成兜底建议（放宽预算/区域/户型）
  7. 7
    确定性评分 score_properties()
    price×0.40 + space×0.20 + facility×0.20 + base×0.20。选 Top 3 附带亮点理由。
  8. 8
    LLM 生成推荐回复
    RECOMMEND_SYSTEM_PROMPT + complete_json。2 个 Few-Shot（结果充足/无结果）。LLM 不可用时降级为规则文本。

🔍 关键细节

  • ✅ 双模式入口：简单条件走 search() 快速管线，复杂查询（通勤/POI/模糊条件）走 search_react() ReAct 循环
  • ✅ 内部做 Filter 提取：EXTRACT_FILTERS_PROMPT（87行 prompt），不依赖 FilterAgent
  • ✅ 内部生成回复：RECOMMEND_SYSTEM_PROMPT（100行 prompt），不依赖 SynthesizerAgent
  • ✅ 渐进放宽 4 级：district → property_type → bedrooms → price×1.2。最终回退英文关键词裸搜
  • ✅ Haversine 地理搜索：从 universities 表取学校坐标，计算球面距离精筛
  • ✅ 通勤多引擎降级：主API→备选→ORS→Haversine 4 级，时间放宽 4 级
  • ✅ 一致性校验：validate_recommendations() 丢弃 LLM 幻觉（不在候选快照中的 ID）
  • ✅ 溯源信息：_build_source_info() 生成 "基于 X 套房源检索 | 条件: 区域:园区" 后缀

📊 CompareAgent — 多维度对比（376行）

C
CompareAgent ✅ 上次是 Stub，现已完整实现
✅ 从 19 行 Stub → 376 行完整实现。评分与解释分离：compute_scores() 确定性计算，LLM 只负责解释。

compare() 执行流程

📋 property_ids
→
📊 _gather_compare_metrics()
→
📐 compute_scores()
→
🤖 LLM 解释
→
📊 七维对比 + 推荐
  1. 1
    加载房源 + POI 通勤数据
    从 PropertyPOI 取 nearest_transit_meters()。从 Review 聚合评分 avg(rating) + count。
  2. 2
    compute_scores() 确定性评分
    4 维度加权：通勤×0.35 + 设施×0.20 + 价格×0.20 + 空间×0.15 + 评价×0.10。支持 3 种优先级：balanced / budget / commute。
  3. 3
    LLM 生成解释（COMPARE_SYSTEM_PROMPT）
    1 个 Few-Shot。要求：基于真实字段，不改分数，每套覆盖 pros/cons/best_for，呼应用户优先级。
  4. 4
    LLM 不可用时降级 _rule_based_compare()
    规则生成 pros/cons/best_for，分数完全不变。输出带 AI_UNAVAILABLE_HINT。

🔍 关键细节

  • ✅ 评分与解释分离：compute_scores() 用真实数据算分，LLM 只能解释不能改分
  • ✅ 3 种优先级：balanced（均衡）/ budget（预算优先）/ commute（通勤优先），权重动态调整
  • ✅ 规则降级：LLM 不可用时规则生成 pros/cons，得分完全一致
  • ✅ 两个入口：compare(property_ids) 直接对比 / compare(cart_agent) 对比购物车全部
  • ✅ compare_scoring 模块：PropertyMetrics + DIMENSION_LABELS + PRIORITY_LABELS，独立可测试
  • ✅ build_dimension_analysis()：生成 7 维 Markdown 分析表（通勤/价格/空间/设施/评价/安全/生活便利）
