① FilterAgent — NL → 结构化条件

①
FilterAgent 1×LLM JSON

从自然语言提取结构化筛选条件，区分硬约束 vs 软偏好

📝 用户消息
→
🤖 LLM complete_json
→
🔧 规则规范化
→
📋 {district, price, amenities…}
  1. 1
    注入上轮条件（如存在）
    将 prev_filters 序列化为 JSON 注入 FILTER_PROMPT 的 {PREV_FILTERS_PLACEHOLDER}
  2. 2
    LLM 单次调用 complete_json
    temperature=0.0, max_tokens=500。Prompt 含 6 个 Few-Shot 示例 + 上下文合并规则
  3. 3
    _normalize_amenities() 规则规范化
    用户口语→标准值映射（"养猫"→"宠物友好"，19条规则）
  4. 4
    返回 AgentResult(data=filters_dict)
    含 hard_filters / soft_preferences 分类，供后续放宽决策使用

🔍 关键细节

  • ✅ 跨轮上下文合并：上轮条件注入 prompt，LLM 判断"增量修改/追加/移除/重置"
  • ✅ 硬约束 vs 软偏好：amenities/room_type/commute=硬约束，price/district/area=软偏好（LLM 判断）
  • ✅ 设施映射表：19 组口语→标准值映射（如 "养猫/养狗"→"宠物友好"）
  • ⚠️ 无 ReAct：handle() 直接覆盖，不走父类 handle_with_react()
  • ⚠️ LLM 不可用=空 JSON：降级策略是返回 {}，不做规则兜底

② SearchAgent — 房源检索 + 渐进放宽

②
SearchAgent 委托型

表面是 ReAct Agent，实际委托 AgentService.recommend_properties() 单体执行

📝 用户消息 + filters
→
🔀 路由判断
→
🗄️ AgentService
→
📦 房源列表 + Top 3
  1. 1
    receive: 用户消息 + 前端 filters + 上轮 extracted_filters
    前端 filters 优先级高于 LLM 提取的 extracted
  2. 2
    extract_filters (LLM)：结构化条件提取
    temperature=0.0, 设施映射 21 条规则, 通勤推断规则, ±10% 价格区间
  3. 3
    _lookup_institution: 学校→坐标
    精匹配 abbreviation → ILIKE name → ILIKE abbreviation，三级回退
  4. 4a
    分支 A：有机构坐标 → _geo_search
    Haversine 精筛，radius 3→5→10→20→50km 渐进放宽
  5. 4b
    分支 B：无坐标 → _search_with_relaxation
    逐级放宽 district→property_type→bedrooms→price_max，英文关键词兜底
  6. 5
    (可选) _filter_by_commute：路线 API 过滤
    多引擎降级链：主API→备选API→ORS→Haversine 兜底。4 级渐进放宽
  7. 6
    _score_properties: 确定性评分
    price×0.40 + space×0.20 + facility×0.20 + base×0.20。选 Top 3
  8. 7
    safe_fallback_check: 安全兜底判断
    结果为空/过少/得分断层→生成兜底建议（放宽预算/区域/户型）
  9. 8
    LLM 生成推荐回复（基于 Top 3）
    RECOMMEND_SYSTEM_PROMPT 含 3 个 Few-Shot（结果充足/偏少/无结果），150-250字口语

🔍 关键细节

  • ⚠️ search_direct() 是 fast path：不走 ReAct，直接委托 AgentService
  • ⚠️ handle() 有 ReAct 壳但未实际使用：P0 重构后全部改为直接执行
  • ✅ 渐进放宽 4 级：district→property_type→bedrooms→price×1.2，保障不空结果
  • ✅ 多引擎通勤降级：主API→备选→ORS→Haversine 4 级
  • ✅ 一致性校验：_validate_recommendations 丢弃 LLM 幻觉（不在候选快照中的 ID）
  • ✅ 评分公式：price×0.40 + space×0.20 + facility×0.20 + base×0.20（纯确定性，非 LLM 评分）

③ SynthesizerAgent — 漏斗阶段语调适配

③
SynthesizerAgent 1×LLM Text

按搜索漏斗阶段（5级）选择不同语气 prompt，单次 LLM 调用生成回复

📊 context.search_state
→
🏷️ 读漏斗阶段
→
🤖 LLM complete_text
→
💬 口语推荐回复
  1. 1
    读取漏斗阶段 (search_state.funnel_stage)
    5 种：explore → calibrate → narrow → compare → decide
  2. 2
    选择对应 STAGE_TONES 语气 prompt
    每阶段有专用语调规则。例如 narrow: "逐套分析匹配度，突出差异化优势，给出明确排序"
  3. 3
    LLM single call complete_text()
    max_tokens=300。口语化中文，1-3句话。无 Few-Shot（prompt 内嵌语调说明）

🎯 5 级漏斗阶段 → 语气

阶段含义语气策略
explore刚开始探索市场快照，引导细化条件
calibrate了解行情解释价位差异，建立预算预期
narrow已明确条件逐套分析，差异化排序
compare最终决策果断推荐，指出 trade-off
decide接近确定确认选择 + 引导下一步

⑥ FAQAgent — 纯规则匹配

⑥
FAQAgent 零 LLM

📝 用户消息

→
⚡ strong/weak/none
→
📋 FAQ 答案 / chips 反问
  1. 1
    match_faq(message) 关键词匹配
    strong: 精确命中（如"押金怎么退"→deposit FAQ）。weak: 模糊命中，返回候选 chips 反问。
  2. 2
    strong → 直接返回预设答案
    含 next_chips（后续建议按钮）+ links（页面深链，如合同签署页）
  3. 3
    weak → 反问确认 chips
    "你想了解的是「押金退还」还是「合同签署」？点按钮选择"
  4. 4
    none → get_faq() 回退匹配
    最终兜底："这是平台使用问题，建议查看帮助中心或联系客服"
