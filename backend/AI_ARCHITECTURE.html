3.3 已知问题

filter_agent prompt 被覆盖 Supervisor._execute_agent("filter_agent") → _run_react_agent() → _build_system_prompt("filter_agent") 使用的是简化版 prompt，而非 filter_agent.handle() 中精心编写的 FILTER_PROMPT（含完整的设施口语映射表+硬约束区分逻辑）。
search_agent 双路径开销 先跑 ReAct loop 生成文本，再无论如何跑一次 AgentService.recommend_properties() 取结构化数据。ReAct 产出只作文本，结构化数据来自旧管线。
synthesizer 不需要 ReAct 走 ReAct loop 但实际无需多轮工具调用，应简化为单次 LLM 调用。

5.3 Intent 枚举（execution_dag.py）

class Intent(str, Enum):
    SEARCH = "search"           # 搜索/推荐（含通勤/POI/行情）
    COMPARE = "compare"         # 房源对比
    MANAGE_CART = "manage_cart" # 购物车管理
    FAQ = "faq"                 # FAQ 政策咨询
    GENERAL = "general"         # 闲聊/兜底

5.4 Intent → DAG 路由逻辑

分类来源：

  1. LLM 可用 → AgentService.classify_message(message, history) 返回 {intent, sub_intent, stage, complexity, routing}
  2. LLM 不可用 → _rule_classify(message) 中文关键词匹配：
    • 找房/通勤/POI/行情关键词 → intent="search"
    • 对比/vs/pk → intent="compare"
    • 购物车/候选/收藏 → intent="manage_cart"
    • 押金/合同/退款 → intent="faq"
    • 默认 → intent="general"

DAG 构建：ExecutionDAG.from_intent(intent) 查表返回对应模板，未知 intent 走 GENERAL 兜底。

6. System Prompt 现状

filter_agent ⚠ 简化版

你是筛选条件提取专家。 你的任务：调用 extract_filters 工具从用户消息中提取结构化搜索条件。 调用一次工具后，**立即用中文输出提取到的条件摘要**，不要再调用其他工具。 输出格式示例："已提取条件：区域=苏州，最高预算=2000元，户型=单间"

⚠ 问题：Supervisor._build_system_prompt() 生成简化版，filter_agent.handle() 中精心编写的 FILTER_PROMPT（含设施口语映射表+硬约束区分）被跳过。

search_agent

你是租房搜索专家。当前漏斗阶段: {stage}。 可用工具（按需使用，不要全部调用）： - extract_filters: 从自然语言提取筛选条件 - property_search: 搜索房源（支持 query + 结构化条件，自动渐进放宽） - score_properties: 对搜索结果进行质量评分 - gap_detect: 检测分数断层 - safe_fallback_check: 检查检索质量 - query_rewrite: 改写模糊查询为精确条件 - poi_lookup: 查询房源周边设施 - commute_calc: 计算房源通勤时间 关键规则： 1. 先调用 extract_filters 从用户消息提取结构化条件 2. 调用 property_search 搜索 3. 有结果 → 调用 score_properties 评分 → **立即用中文输出推荐回复** 4. 无结果 → 调用 safe_fallback_check → **立即用中文告知用户并给出放宽建议** 5. **最多调用 3 个工具**，之后必须输出中文文本回复 6. 回复格式：先总结匹配结果数量，再逐套介绍 7. 如果用户提到了通勤需求，可额外调用 commute_calc 8. 如果用户提到了周边配套需求，可额外调用 poi_lookup

synthesizer_agent

你是租房对话回复合成专家。当前漏斗阶段: {stage}。 将多个 Agent 的分析结果融合为自然、友好的中文回复。 - explore阶段：市场概览语气，引导用户细化需求 - calibrate阶段：比较性语气，帮用户理解价格与质量的对应关系 - narrow阶段：分析性语气，对比区域差异 - compare阶段：决策性语气，给出明确推荐 - decide阶段：行动导向，引导下一步（预约看房等） 如果需要检查检索质量，可以调用 safe_fallback_check 和 build_fallback_reply。 完成任务后必须直接输出中文文本回复。

compare_agent / cart_agent / faq_agent

这三个 Agent 不使用 LLM，无需 System Prompt。它们直接委托现有 Service 层方法：

  • compare_agent → AgentService.compare_cart()
  • cart_agent → AgentService CRUD 方法
  • faq_agent → agent_faq.match_faq() 规则匹配

注：ReAct loop 仍会为它们构建兜底 prompt（"你是 {agent_name}。完成分配给您的任务。" + Handoff 指令），但实际执行不会走 ReAct 路径（Supervisor._execute_agent 对这三个直接调用专用 handler）。

⚠ 所有 Agent 共享的 Handoff 指令

每个 Agent 的 System Prompt 末尾都追加了 Handoff 指令块，列出其他可用 Agent 并告知可通过 handoff_to() 工具交接任务。Handoff 链模式（handle_message_handoff）是备用执行路径，当前 /messages 端点不使用。

核心结论

所有搜索子场景（通勤/POI/行情）统一走 SEARCH DAG，不再需要独立 Agent。

下一步重点：移除核心 Agent 的 ReAct 开销（filter/synth 不需要多轮工具调用），重写 prompt 加入漏斗阶段语境。
