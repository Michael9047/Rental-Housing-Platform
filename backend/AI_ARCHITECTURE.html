3.2 关于 MoE 专家组的具体论证

MoE（Mixture of Experts）的设计前提：有大量候选房源（>20套），需要从多个独立维度深度分析，每个维度需要独立的 LLM 视角，最后加权融合。

为什么学生租房不需要：

  • 候选集通常只有 5-15 套，不需要 5 个专家并行分析
  • 学生租房决策因素简单：价格 > 通勤 > 独卫 > 配套。不需要"空间利用率""区域发展潜力"这种维度
  • 5 个专家并行调 LLM = 5× 的 token 消耗 + 5× 的延迟，换来的只是"空间专家说这个户型还不错"——用户不关心
  • area_expert 在没有犯罪数据的情况下分析"安全性"，纯 LLM 编造
  • EstateWise 的 MoE 是为美国买房场景设计的（几十万美金决策需要多维度尽调），跟中国学生租房完全不搭

设施硬约束怎么办？ amenity_expert 的 AND 语义检查（必须同时满足独卫+阳台+WiFi）是纯 Python 逻辑，不需要 Agent。把这个检查逻辑放到 search_agent 的评分阶段就行——不满足硬约束的直接从候选集排除。

3.3 关于 market_agent 的具体论证

market_agent 的设计意图：分析市场价格趋势、投资回报率、同比涨跌。

为什么学生租房不需要独立 Agent：

  • 学生租房周期 3-12 个月，价格在这个时间窗口内基本不变
  • 用户问"园区单间大概多少钱"需要的是当前快照（DB 聚合查询），不是时间序列分析
  • market_stats 工具已经能返回价格分布/户型分布，synthesizer_agent 直接把这些数据转成自然语言描述即可
  • 分层卡片（经济型/舒适型/高端型）是前端渲染 + DB 聚合的事，不需要 Agent 参与
  • 如果未来需要真正的趋势分析（同比/环比），在 market_stats 工具里加就好，仍然不需要独立 Agent

6.3 建议的 SEARCH DAG（精简为一条主链路）

不再区分 SEARCH / COMMUTE_INFO / POI_INFO / MARKET_ANALYSIS 四种意图。统一为一条 DAG，按需启用可选层。

[Layer 0] filter_agent: LLM JSON提取筛选条件（单次调用，不走ReAct） │ [Layer 1] search_agent: property_search + score_properties + gap_detect │ （走旧管线 recommend_properties，确定性评分，不需要 ReAct） │ [Layer 2] ┌─ commute_calc（DB查，如果有通勤需求） （并行） └─ poi_lookup （DB查，如果有POI需求） │ [Layer 3] synthesizer_agent: 单次 LLM 调用合成回复

Layer 0 ← 规则触发（消息含筛选关键词）
Layer 1 ← 始终执行
Layer 2 ← 条件触发（消息含通勤/POI关键词 或 显式要求）
Layer 3 ← 始终执行
全程只有 1-3 次 LLM 调用（filter 1次 + search 0次 + synth 1次 + 可选并行层 0次），无 ReAct 开销。

7. System Prompt 现状与改造方向

filter_agent 的 prompt 问题

Supervisor 用的是简化版，filter_agent 自己精心写的 FILTER_PROMPT（含设施口语映射表+硬约束区分）被跳过了。

修复：让 filter_agent 走自己的 handle() 方法（单次 LLM JSON 提取），不走 _run_react_agent()。

synthesizer 的 prompt 改造

加入学生租房漏斗阶段的语境引导：

你是西交利物浦大学周边的租房顾问。 当前阶段: {stage} - explore: 用户刚开始看，给市场概览，引导细化需求 - narrow: 用户有明确条件，逐套分析匹配度 - compare: 用户在做决定，给出明确推荐和理由 - decide: 用户即将预订，引导下一步（预约看房） 回复要求：中文口语化，2-3句话，不要编造数据。
