# AI 租房助手——用户场景覆盖分析

> 目标：明确 AI Agent 需要处理哪些租房相关问题，据此设计 Agent 层和工具层。

---

## 第一层：核心搜索（日常最高频） 🔴

预算帽 + 区域限定 + 硬约束需求。例如：独立卫浴、半小时地铁到学校、周边有商场。

这是最主要调用的场景，涉及数据库读取和 API 工具调用。也是优先开发的核心链路。

### 场景 1：预算帽 + 区域 + 硬约束搜索

**用户说什么：**
- "园区 2000 以内，独卫单间，到西交利物浦地铁半小时"
- "独墅湖高教区 2500 以下，合租也行但要独立卫浴，最好附近有超市"
- "学校附近 1500-2000，要有空调和 WiFi，接受合租"

**需要的能力：**

| 步骤 | 工具 | 数据来源 |
|------|------|---------|
| 提取筛选条件 | LLM NL→JSON（区域/预算/户型/设施/通勤） | DeepSeek |
| 结构化搜索 | property_search（向量 + 条件筛选） | PostgreSQL |
| 通勤计算 | commute_calc（房源 → 学校，公交/步行/骑行） | 详见第四节数据策略 |
| POI 查询 | poi_lookup（周边超市/地铁/商场） | PostgreSQL POI 表 |
| 质量评分 | score_properties（价格+空间+设施+地段） | 纯算法 |
| 合成推荐 | LLM 生成中文推荐回复 | DeepSeek |

**DAG 流程：**
```
filter_agent → search_agent(含评分) → [可选: commute_agent + poi_agent] → synthesizer_agent
```

### 场景 2：自然语言相对查询（迭代缩小/扩大）

**用户说什么：**
- "太贵了，便宜一点的"
- "有没有离学校更近的"
- "大一点的呢"
- "还有别的吗"

**需要的能力：**
- `query_rewrite` 工具：理解相对描述，调整筛选参数（"便宜一点"→budget×0.8，"更近"→commute 权重↑）
- 保持搜索上下文（search_state），知道上一轮的结果
- 多样性注入：避免每次推荐同一批房源

### 场景 3：条件缺失时的引导

**用户说什么：**
- "帮我找房子"
- "学校附近"

**需要的能力：**
- 识别信息缺口（缺预算？缺户型？缺区域？）
- 反提问引导（"您预算大概多少？需要独卫吗？"）
- 先给大范围概览再逐步缩小（而非直接报错）

### 场景 4：特定偏好搜索

**用户说什么：**
- "可以养猫的"
- "有阳台的，最好朝南"
- "精装修拎包入住的"
- "包水电的"

**需要的能力：**
- filter_agent 已有设施口语映射表（宠物友好/独立厨房/独卫/WiFi/车位/阳台...）
- 需要确认数据库 `property` 表有对应的字段或标签来支持这些筛选条件

## 第二层：区域行情分析（帮助用户校准预期） 🟡

基于某学校附近，房源质量大概都有哪些层次。通过卡片的形式展示不同评分、配套、地段层次的公寓，让用户快速理解行情，避免预算和实际市场脱节。

### 场景 5：快速了解某区域的房源层次

**用户说什么：**
- "西交利物浦附近租房大概什么价位？"
- "园区单间一般多少钱？"
- "独墅湖那边有什么类型的房子？"

**需要的输出形式：**
- **分层卡片**：例如"园区单间分为三档——经济型(1000-1500,配套一般,距校>20min)/舒适型(1500-2500,独立卫浴,配套齐全)/高端型(2500+,精装修,紧邻地铁)"
- **关键指标分布**：价格分布直方图、户型分布、面积分布
- **代表房源**：每档展示 1-2 套典型房源

**需要的能力：**

| 步骤 | 工具 | 说明 |
|------|------|------|
| 区域统计 | market_stats（价格分布/户型分布/面积分布） | 从 DB 聚合 |
| 分层聚类 | 按价格+评分+面积自动分档 | 需要实现分档算法 |
| 代表采样 | 每档取高分房源 | property_search + 评分 |
| 可视化 | 前端渲染分层卡片 | 前端工作 |

**DAG 流程：**
```
filter_agent → search_agent(拿候选集) → market_agent(统计+分层) → synthesizer_agent(生成分层描述)
```

### 场景 6：预算可行性判断

**用户说什么：**
- "园区 1000 块能租到独卫单间吗？"
- "1500 预算在独墅湖够不够？"

**需要的能力：**
- 市场统计 → 判断用户预算在该区域的 percentile
- 如果预算过低，给出"最近匹配的房源也要 XXX 元"或"可以考虑合租"的建议
- 如果预算充裕，告诉用户"这个预算可以租到舒适型甚至高端型"

## 第三层：多维度对比分析 🟡

将候选清单中的房源按用户需求进行设施、配套、评分、地段、通勤等多维分析对比，每一维度进行横向比较分析。

### 场景 7：候选清单多维度对比

**用户说什么：**
- "帮我对比购物车里的这三套"
- "哪个性价比最高？"
- "综合考虑通勤和价格，哪个更合适？"

**需要的输出形式：**

| 维度 | 房源A | 房源B | 房源C | 分析 |
|------|-------|-------|-------|------|
| 月租 | 1800 | 2200 | 2000 | A最便宜 |
| 面积 | 25㎡ | 35㎡ | 28㎡ | B最大 |
| 到校时间 | 15min | 30min | 20min | A最近 |
| 周边配套 | 超市+地铁 | 商场+健身房 | 超市+餐馆 | B配套最丰富 |
| 独立卫浴 | ✅ | ✅ | ❌ | C不合要求 |
| 综合评分 | 78 | 82 | 65 | B综合最优 |

**综合推荐：** 每套房的适用场景总结（"A适合预算敏感/通勤优先"、"B适合追求品质"）

**需要的能力：**
- `compare_dimensions` 工具：多属性矩阵，每维度横向比较
- 按用户优先级加权（"我主要看重通勤"→通勤维度权重↑）
- 价格/通勤/空间/评价四个维度各自评分 + 综合加权

### 场景 8：快速二选一

**用户说什么：**
- "A和B哪个好？"
- "这两套选哪个？"

**需要的能力：**
- 简化版对比（单轮对话，不需要完整对比报告），直接给出建议 + 简短理由

## 第四层：数据策略——查询时计算 → 摄入时预计算 🔵

当前架构中，通勤时间、POI 丰富度等信息在用户查询时实时调用 API 计算。这有几个问题：
- **慢**：每次查询都要等待外部 API 响应
- **贵**：频繁调用高德/OSRM 产生费用
- **脆**：外部 API 不可用时搜索直接失败
- **不准**：用户搜索时间（比如晚上）的实时路况不代表日常通勤时间

**核心思路：在房源入库时就把能算的都算好，存入数据库。查询时只读不调 API。**

### 4.1 通勤时间预计算

**当前状态：** `commute_calc` 工具在查询时实时调用高德/OSRM API

**改为：** 房源入库（或更新坐标）时，后台异步计算到附近热门目的地的通勤时间，写入 `property_commutes` 表。

```
property_commutes 表结构：
  property_id    → 房源 ID
  destination    → 目的地标识 (如 "xjtlu", "suda", "sip_central")
  destination_name → 中文名 (如 "西交利物浦大学")
  mode           → transit / walking / bicycling / driving
  duration_min   → 通勤分钟数
  distance_km    → 距离公里
  computed_at    → 计算时间（数据新鲜度追踪）
```

**热门目的地预定义：**
- 西交利物浦大学
- 苏州大学独墅湖校区
- 园区主要商圈（苏州中心、圆融时代广场）
- 主要地铁站（1号线/2号线/3号线各站）

**Agent 查询时：** `commute_calc` 不再是调 API，而是：
```sql
SELECT * FROM property_commutes
WHERE property_id IN (:candidate_ids)
  AND destination = :destination
  AND mode = :mode
  AND duration_min <= :max_minutes
```
纯数据库查询，毫秒级。

**更新策略：**
- 房源新增 → 异步计算所有预设目的地
- 房源坐标变更 → 标记 computed_at 过期，重新计算
- 定时任务 → 每月全量刷新一次（路况/公交线路可能变化）

### 4.2 POI 周边设施预计算

**当前状态：** `poi_lookup` 工具读 POI 表

**确认/加强：** 房源导入时就调高德周边搜索 API，把周边 POI 写入 `poi` 表。查询时只读。

```
poi 表（已有，确认字段）：
  property_id  → 房源 ID
  poi_data     → JSON: {
      "supermarket": [{"name":"盒马","distance":300}, ...],
      "subway": [{"name":"月亮湾站","distance":500}, ...],
      "restaurant": [...], "gym": [...], "hospital": [...]
    }
  updated_at   → 数据新鲜度
```

**POI 类型（学生租房场景重点）：**
- 交通：地铁站、公交站
- 生活：超市、便利店、菜市场
- 餐饮：餐馆、食堂
- 学习：打印店、自习室、图书馆
- 健康：健身房、医院/诊所

### 4.3 区域安全/环境信息

**当前状态：** 无。`area_expert` 的 prompt 写了"分析安全性"但没有数据源，纯 LLM 编造。

**改为：** 建立 `area_profiles` 表，存储区域级别的元数据。数据来源可以是：
- 政府公开数据（治安通报、社区评级）
- 平台自有数据（用户评价中的安全相关反馈）
- 如果暂时没有外部数据源，先手工录入区域描述（学生区/商业区/工业区等客观事实，不做主观安全评分）

```
area_profiles 表结构：
  district      → 区域名（如"工业园区独墅湖"）
  profile       → JSON: {
      "type": "学生区",           // 学生区/商业区/工业区/住宅区
      "noise_level": "低",        // 低/中/高（基于区域类型推断）
      "student_friendly": true,   // 是否有学生聚集
      "night_activity": "安静",    // 安静/一般/热闹
      "description": "独墅湖高教区核心，西交利物浦/苏大周边，学生为主"
    }
  source        → 数据来源（"manual" / "gov_open_data" / "user_reviews"）
  updated_at
```

**注意：** 没有权威犯罪数据源时，不要编造安全评分。用客观事实（区域类型、学生比例）替代主观评价（"安全"），让用户自己判断。

### 4.4 数据策略对 Agent 架构的影响

**之前（查询时计算）：**
```
用户搜索 → Agent 调 commute_calc → 等 API(2-5s) → 调 poi_lookup → 等 DB(100ms) → 评分 → 回复
总耗时：3-6 秒
```

**之后（预计算+DB 读）：**
```
用户搜索 → Agent 调 commute_calc → 查 DB(10ms) → 调 poi_lookup → 查 DB(10ms) → 评分 → 回复
总耗时：<1 秒
```

**Agent 工具层不需要变**——`commute_calc` 和 `poi_lookup` 的 handler 签名不变，只是内部实现从"调外部 API"变成"查数据库"。Agent 无感知。

**需要新增的模块：**
- 房源导入后的异步计算任务（Celery 或后台线程）
- `area_profiles` 表 + 管理后台录入界面
- 定时刷新任务（每月更新通勤和 POI）
