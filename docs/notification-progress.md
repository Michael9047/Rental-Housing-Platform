> 更新日期：2026-07-06

## 一、微信通知未实现的原因

### 1.1 企业微信机器人

企业微信机器人（Webhook）虽然接入简单，但存在根本性局限——**消息接收方必须是企业微信好友（同企业通讯录成员）**。对于租房平台而言，租客和房东不在企业通讯录中，因此企业微信机器人无法用于 C 端用户触达，仅可服务于内部 BD 经理或管理员的系统告警。

### 1.2 小程序订阅消息

微信小程序订阅消息是 C 端推送的最佳渠道，但其**强依赖小程序本身的开发与上线**：

- 订阅消息需要用户在小程序端主动点击授权，后端才能推送；
- 模板 ID 需要在微信小程序后台申请并审核通过；
- 用户的 openid 只能通过 wx.login() 在小程序端获取，网页端用户无法直接获得。

同时，网页端手机号注册的用户 **不会自动获得 wechat_openid**，无法触发微信推送。需要额外做"网页引导扫码关注→绑定 openid"的链路，才能在非小程序场景下建立微信触达能力。

> **结论：微信通知（小程序订阅消息）需等待小程序开发完成后才能正式启用。建议同一消息走 Email + 微信双通道，Email 保证覆盖率，微信提升触达率。**

## 二、当前消息通知实现情况

### 2.1 整体架构

`
业务触发 (booking/payment/contract/...) 
    → NotificationService.create_notification()
        → 写入 notifications 表（站内通知）
        → _dispatch_channels()
            ├── Email   → send_email_notification (Celery)
            ├── SMS     → send_sms_notification (Celery) [仅验证码]
            └── WeChat  → 暂未启用
`

### 2.3 当前通知的适用对象

目前消息通知体系主要面向**租客（C 端用户）**，覆盖看房预约、支付、合同签署等核心流程中的状态变更通知。房东端和管理员端的通知暂未独立推送，相关状态变更通过站内通知统一记录。

后续 App 端开发阶段，计划扩展以下通知场景：

| 计划场景 | 目标对象 | 说明 |
|----------|----------|------|
| 维修工单通知 | 租客 ↔ 房东 ↔ 维修人员 | 报修提交、派单、完工确认 |
| 新房源预定通知 | 租客 | 关注的房源上线 / 降价提醒 |
| 房东端业务通知 | 房东 | 新预约提醒、房源审核结果、租金到账 |
| BD 经理任务通知 | 管理员 / BD | 房源审核任务、异常订单告警 |
| App 原生推送 | App 用户（租客 + 房东） | iOS APNs / Android FCM，锁屏级通知，与 Email / 微信互补 |

## 三、SMS 业务通知不可用的原因

SMS 发送可编辑的长消息通知需要以下完整链路：

`
模板 (Template) → 签名 (Signature) → 委托授权书 → 资质审核 → 企事业单位信息
`

具体依赖如下：

| 环节 | 所需材料 | 当前状态 |
|------|----------|:--------:|
| **资质申请** | 企业营业执照、法人身份证、管理员身份证 | ⚠️ 无企业主体 |
| **短信签名** | 委托授权书（如签名名称为品牌名，需品牌方授权） | ❌ 依赖资质 |
| **短信模板** | 提交模板内容，运营商审核（1-3 工作日） | ❌ 依赖签名 |
| **发送消息** | 调用阿里云 SMS API，传入模板参数 | ❌ 依赖模板 |

当前项目的短信实现（SmsService）使用的是阿里云**号码认证**（dypnsapi）的 SendSmsVerifyCode 接口，这是一个专用验证码通道，模板已由阿里云预设（"您的验证码为。分钟内有效"），无需自建签名和模板。但要发送自定义内容的业务通知（如"您的预约已确认"），必须走标准 SMS 通道，这就回到了上面的审核链路。

> **一句话：没有企业营业执照就无法申请签名，没有签名就无法创建模板，没有模板就不能发自定义内容的短信。这是运营商监管的硬性要求，不是技术问题。**

## 四、建议触达策略

采用**分渠道分层触达**方案，各渠道承担不同的职责：

`
┌───────────┬──────────────────────┬──────────────┐
│   渠道     │       职责           │  覆盖用户     │
├───────────┼──────────────────────┼──────────────┤
│ Email     │ 所有业务通知（长消息）  │ 所有注册用户  │
│ SMS       │ 验证码（登录/注册）     │ 所有手机号用户│
│ 微信       │ 全量业务通知（双通道）  │ 小程序用户   │
│ 站内通知   │ 所有通知记录留存       │ 所有注册用户  │
└───────────┴──────────────────────┴──────────────┘
`

**核心策略：**

- **Email 为主力**：承担所有可编辑长消息的业务通知（预约、支付、合同等），无内容限制，已完整实现；
- **SMS 聚焦验证码**：不追求业务通知，专做登录注册验证码，这是 SMS 能力边界内无需额外审核的部分；
- **微信双通道**：小程序上线后，微信用户的所有通知同时在 Email 和微信两端触达，构成双重保障——Email 确保送达，微信提升即时感知；
- **站内通知兜底**：所有消息持久化在 
otifications 表中，前端轮询或 WebSocket 推送，确保即使外部渠道失败用户也能在平台内看到。

### 4.1 实现优先级

| 优先级 | 事项 | 依赖 |
|:------:|------|------|
| P0 | Email 通知全部 14 种类型 | ✅ 已完成 |
| P1 | SMS 验证码（登录/注册） | ✅ 已完成 |
| P2 | 微信小程序订阅消息 | 需小程序上线 + 模板审核 |
| P3 | 企业微信机器人（内部告警） | 需注册企业微信 + 配置 Webhook |
| P4 | SMS 业务通知模板 | 需企业资质（营业执照等） |
