The repository is a flat monorepo whose children are independent deployable units wired together at the root level:
- `docker-compose.yml` (dev) and `docker-compose.prod.yml` (prod) define shared infrastructure — PostgreSQL 16 with pgvector and Redis 7 — plus the production stack of `backend`, `celery-worker`, `celery-beat`, and an Nginx reverse proxy that serves the built `frontend` SPA while proxying `/api/v1` to the backend. The three-tier network (`frontend` ↔ `backend` ↔ `data`) enforces isolation between the web tier, API tier, and data tier.
- `.env.example` / `.env.prod` provide the single source of truth for all cross-cutting configuration (DB URLs, Redis URL, OpenAI keys, WeChat AppID/Secret, CORS origins), consumed by both the backend service and the prod compose file.
- `run_dev.ps1` is the Windows one-shot dev entry point: it brings up the infra containers via docker-compose, then launches the backend's uvicorn server and the frontend's Vite dev server in parallel, giving developers a single command to run the full stack locally.
- `design_system` and `docs` are static artifacts consumed by code generators and team workflow; they have no runtime wiring but live alongside the code so generators can reference them from sibling directories.
