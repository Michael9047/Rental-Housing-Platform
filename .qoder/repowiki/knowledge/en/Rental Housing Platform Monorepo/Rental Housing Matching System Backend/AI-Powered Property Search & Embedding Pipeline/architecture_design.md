Two parallel concerns share this scope:

1. AI search API (`routes/ai_search.py` + `schemas/ai_search.py`) — a FastAPI `APIRouter` exposing `/parse` (LLM-based query parsing into `ParsedSearchParams` + `CompletenessReport`) and `/search` (hybrid retrieval via `PropertyService.search`, then LLM summary over top-3 results). The route layer is thin: it builds the search query string, delegates to `PropertyService`, maps ORM `Property` rows to `PropertySearchResult` / dict payloads, and wraps LLM calls in try/except blocks that degrade gracefully when no LLM is configured.

2. Embedding pipeline (`models/embedding_job.py` + `services/embedding_service.py` + `tasks/embedding_tasks.py` + `services/embedding_job_service.py`) — `EmbeddingJob` models a long-running job per property with a four-state status machine (`pending → processing → completed | failed`). `EmbeddingService` wraps `AsyncOpenAI` using settings from `get_settings()` to call the configured embedding model. Celery tasks `generate_property_embedding` and `reindex_all_properties` spin up their own async SQLAlchemy engine/session per task invocation, create a pending job row, mark processing/completed/failed with timestamps, persist the resulting vector on `Property.embedding`, and raise so Celery's `autoretry_for=(Exception,)` / `max_retries=3` can retry. `EmbeddingJobService` provides list/stats/reindex triggers backed by `select`/`func.count` queries and enqueues the Celery tasks via `.delay()`.

Dependency direction: routes → services/schemas; services → models + external clients (OpenAI, LLM); tasks → services + models + Celery app; schemas are pure Pydantic types consumed by both routes and responses.
