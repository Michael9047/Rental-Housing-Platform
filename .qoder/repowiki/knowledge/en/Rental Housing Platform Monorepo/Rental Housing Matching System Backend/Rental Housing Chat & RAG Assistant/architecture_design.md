Three-layer leaf inside the app package:
- `models/chat.py` defines SQLAlchemy 2.0 declarative models (`ChatSession`, `ChatMessage`) with `TimestampMixin`, PostgreSQL `JSON` for message metadata, and `Enum` columns for status/role; sessions own a `selectin` relationship to messages with cascade delete.
- `services/chat_service.py` is the single orchestration class: session CRUD, an async `_build_rag_context` that lazily imports `EmbeddingService` + `pgvector.sqlalchemy.l2_distance` to search `Property.embedding` vectors (top-5 available listings), builds an OpenAI prompt via `_build_messages` (system prompt in Chinese plus matched properties), then calls `AsyncOpenAI.chat.completions.create`. Two entry points exist — `chat()` returns a dict, `chat_stream()` yields SSE frames of type `matched` / `content` / `done` / `error`.
- `api/v1/routes/chat.py` exposes REST endpoints under `/sessions` and `/sessions/{id}/messages`; each handler constructs a `ChatService(session)` from the injected `AsyncSession` and current user, maps ORM rows to Pydantic v2 schemas (`SessionResponse`, `MessageResponse` with `from_attributes`), and wraps the streamer in FastAPI's `StreamingResponse(text/event-stream)` with no-cache headers.
Dependency direction is strictly routes → service → models; the service also reaches out to `app.models.property.Property` and `app.services.embedding_service.EmbeddingService` (imported lazily) while staying unaware of HTTP concerns.
