Layered FastAPI resource split across two sibling routers: `routes/properties.py` exposes `/api/v1/properties` (CRUD + `/search`) and `routes/images.py` exposes nested `/api/v1/properties/{id}/images` (upload, list, delete, set-primary). Routes are thin — they only handle dependency injection (`get_db_session`, `require_landlord`), request validation via Pydantic schemas, authorization checks against `current_user.role`, and delegation to service classes. `services/property_service.py` encapsulates all DB logic (create/list/search/update/delete) plus cross-cutting concerns: a lazy-imported async Redis cache keyed by `search:{prefix}:{json(params)}` with a 300s TTL for non-vector queries, an async POI enrichment call via `POIService`, and a background Celery task dispatch (`generate_property_embedding.delay`) launched from a daemon thread. `services/image_service.py` owns filesystem I/O under `settings.upload_dir`, writing files named `{uuid4().hex}{ext}` and enforcing per-property image count/type/size limits read from `core.config.get_settings()`. Models in `models/property.py` and `models/property_image.py` use SQLAlchemy 2.0 `Mapped`/`mapped_column` with `TimestampMixin`; the `embedding` column is a custom `VectorColumn` TypeDecorator that switches between `pgvector.sqlalchemy.Vector(1536)` on PostgreSQL and plain `Text` otherwise. Schemas in `schemas/` mirror this split: `PropertyBase` / `PropertyCreate` / `PropertyUpdate` / `PropertyRead` / `PropertySearchResult` (with a `primary_image_url` computed property) and `PropertyImageRead`. Dependency direction is strictly routes → services → models; services may reach into `app.services.poi_service`, `app.services.embedding_service`, and `app.tasks.embedding_tasks` but never back into routes.
